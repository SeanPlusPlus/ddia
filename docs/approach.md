# Study Approach

## Method

1. **Read** the chapter start to finish, no notes
2. **Debrief** — talk through what stood out, what clicked, what felt fuzzy. Claude writes the structured notes from the conversation.
3. **Review** — edit the notes, add anything missed, commit
4. **Apply** — connect concepts to real systems I've built or operated (inline in the notes or in `applied/`)
5. **Question** — capture what's still unclear or what I'd challenge

## Note Template

Each chapter note follows this structure:

```markdown
# Chapter N: Title

## TL;DR
One paragraph summary.

## Key Concepts
- Concept → explanation → real-world mapping

## Diagrams
Mermaid diagrams for important data flows or architectures.

## Applied
How this maps to systems I work with (AWS, CDN, caching, agents).

## Open Questions
What I'd dig deeper on, disagree with, or want to discuss.
```

## Why DDIA

DDIA is the canonical text for understanding the internals of the systems we use daily — databases, message queues, stream processors, consensus protocols. The 2nd edition updates coverage for the modern landscape. As someone building production distributed systems on AWS, this deepens the mental model behind the abstractions.
